# خطة تجهيز مشروع المنصة قبل النشر

> اسم المشروع الحالي: منصة شركة برمجيات / موقع عام + لوحة تحكم + Backend API  
> حالة هذا الملف: خطة عمل داخلية لتجهيز المشروع بالكامل قبل مرحلة النشر  
> آخر تحديث: 2026-06-13  
> ملاحظة مهمة: هذه الخطة ليست خطة نشر إنتاجي الآن. هدفنا الحالي هو إغلاق المشروع برمجيًا ووظيفيًا وتجهيزه، ثم لاحقًا ننتقل إلى النشر.

---

## 1. الهدف الحالي

الهدف الآن ليس رفع الموقع على سيرفر إنتاجي، بل الوصول بالمشروع إلى نسخة قوية ومستقرة يمكن الاعتماد عليها لاحقًا عند النشر.

الأولوية الحالية:

1. تثبيت بنية المشروع.
2. التأكد أن لوحة التحكم تحفظ وتعدل كل البيانات بشكل صحيح.
3. التأكد أن الموقع العام يعرض البيانات الحقيقية من الباكند بدون مشاكل.
4. إصلاح أي خلل في الـ API أو الـ Serializers أو المصادقة أو الصلاحيات.
5. تنظيف الملفات والكاش والملفات غير الضرورية.
6. تجهيز اختبارات وفحص جودة قبل اعتماد النسخة.
7. بعد الإغلاق الكامل فقط نبدأ بمرحلة النشر.

---

## 2. البنية الحالية للمشروع

تم تنظيم المشروع إلى ثلاث وحدات مستقلة من جذر المشروع:

```text
platform/
├─ backend/       # Django REST API + قاعدة البيانات + endpoints عامة وخاصة
├─ public_site/   # الموقع العام للزوار والعملاء
├─ admin_panel/   # لوحة التحكم الداخلية
├─ docs/          # توثيق مراحل العمل والتقارير
├─ deployment/    # ملفات النشر لاحقًا، وليست أولوية الآن
└─ README.md
```

هذا التنظيم صحيح ومناسب لمشروع احترافي لأن كل جزء منفصل وظيفيًا:

- `backend` مسؤول عن البيانات والمنطق والـ API.
- `public_site` مسؤول عن تجربة الزائر والعميل.
- `admin_panel` مسؤول عن إدارة محتوى الموقع والمنصة.

## 4. المشاكل والملاحظات التي ظهرت أثناء الفحص

### 4.1 المشروع ليس في مرحلة نشر الآن

لا نريد الآن تعديل Nginx أو تجهيز السيرفر أو الدومين. هذه الأمور مؤجلة.

لكن يجب ترك ملفات النشر موجودة كتوثيق لاحق، مع عدم اعتبارها أولوية حالية.

الحالة: قرار واضح — النشر لاحقًا.

---

### 4.2 مشكلة CI في GitHub Actions

في ملف:

```text
.github/workflows/ci.yml
```

يوجد أمر قديم يفترض وجود مجلد `app`:

```bash
python -m compileall app tests
```

بينما البنية الحالية تستخدم:

```text
config/
platform_api/
scripts/
tests/
```

المطلوب لاحقًا:

```bash
python -m compileall config platform_api scripts tests
```

الأولوية: عالية.

---

### 4.3 إنشاء المستخدمين وكلمات المرور

تم ملاحظة أن صفحة المستخدمين في لوحة التحكم قد ترسل `password`، بينما الباكند يتعامل مع `hashed_password` أو لا يعالج كلمة المرور بالشكل المطلوب.

المطلوب:

- إصلاح `UserSerializer`.
- قبول `password` عند إنشاء مستخدم جديد.
- تمرير كلمة المرور عبر `hash_password()`.
- عدم إرجاع كلمة المرور في أي Response.
- إضافة endpoint أو منطق آمن لتغيير كلمة المرور.

الأولوية: حرجة.

---

### 4.4 حفظ البيانات المتداخلة داخل الخدمات والأنظمة والتطبيقات

هناك احتمال أن بعض الحقول المتداخلة لا يتم حفظها لأن بعض الـ Serializers فيها `read_only=True`.

أمثلة على بيانات يجب التأكد من حفظها:

- Features
- Images
- FAQs
- Screenshots
- Download links
- Product features
- Service features

المطلوب:

- فحص كل فورم في لوحة التحكم.
- التأكد من أن البيانات تصل للباكند.
- التأكد من أن الباكند يحفظها.
- التأكد من أن الموقع العام يعرضها.
- إما إضافة `create/update` مخصص داخل Serializer، أو إنشاء endpoints منفصلة لإدارة البيانات المتداخلة.

الأولوية: حرجة.

---

### 4.5 استجابات الأخطاء في API

بعض الاستجابات قد ترجع حالة HTTP خطأ مثل 401 أو 404، لكن جسم الاستجابة يحتوي:

```json
{
  "success": true
}
```

المطلوب:

- أي خطأ يجب أن يرجع:

```json
{
  "success": false
}
```

- توحيد شكل الأخطاء في كل الباكند.
- التأكد أن الواجهة تتعامل مع الخطأ بشكل صحيح.

الأولوية: عالية.

---

### 4.6 ملفات ضخمة تحتاج تقسيم لاحقًا

بعض الملفات أصبحت كبيرة، مثل:

```text
AdminAppsPage.tsx
AdminProductsPage.tsx
AdminServicesPage.tsx
AdminSettingsPage.tsx
backend/platform_api/views.py
backend/platform_api/models.py
```

هذا لا يمنع التشغيل، لكنه يقلل سهولة الصيانة.

المطلوب لاحقًا:

- تقسيم صفحات الأدمن إلى:
  - components
  - hooks
  - forms
  - tables
  - modals
  - api helpers
- تقسيم الباكند إلى ملفات أكثر تخصصًا عند الحاجة.

الأولوية: متوسطة.

---

### 4.7 تخزين الجلسة في localStorage

حاليًا تخزين التوكن في `localStorage` مناسب للتطوير، لكنه ليس الخيار الأفضل للإنتاج.

المطلوب لاحقًا قبل النشر:

- دراسة نقل المصادقة إلى HttpOnly Secure Cookies.
- أو تشديد الحماية عبر CSP وتقصير عمر التوكن.

الأولوية الآن: متوسطة.  
الأولوية قبل النشر: عالية.

---

### 4.8 تنظيف النسخة الخفيفة

النسخة الخفيفة جيدة، لكن يجب تحسين أمر النسخ حتى يحذف أيضًا:

```text
__pycache__/
*.pyc
*.pyo
*.tsbuildinfo
.pytest_cache/
.mypy_cache/
.ruff_cache/
```

الأولوية: متوسطة.

## 7. قائمة ممنوعات قبل إغلاق المشروع

قبل الانتهاء من التجهيز، ممنوع اعتبار المشروع جاهز إذا بقي أي شيء من التالي:

- فورم في لوحة التحكم لا يحفظ فعليًا.
- زر موجود لكنه لا يعمل.
- صفحة عامة فارغة رغم وجود بيانات.
- خطأ TypeScript.
- خطأ ESLint.
- خطأ Build.
- فشل في إنشاء مستخدم.
- كلمة مرور تحفظ بدون تشفير.
- API يرجع `success: true` مع خطأ HTTP.
- ملفات `.env` حقيقية داخل النسخة.
- وجود `node_modules` أو `.next` أو uploads أو database داخل نسخة الرفع.

المشروع حاليًا يعتبر:

```text

```

وليس:

```text
جاهز نشر إنتاجي
```

التقييم التقريبي الحالي:

- جودة البنية: جيدة جدًا.
- فكرة المشروع: قوية.
- لوحة التحكم: متقدمة وتحتاج تثبيت حفظ البيانات.

- الباكند: جيد كبنية لكنه يحتاج إصلاحات حرجة.
- النشر: مؤجل حاليًا.

الهدف القادم:

```text
الوصول إلى نسخة مغلقة ومستقرة محليًا بنسبة 95% قبل التفكير بالنشر.
```
